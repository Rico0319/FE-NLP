Slide 1: 10-K Data Extraction & NLP Pipeline

SEC EDGAR API: Downloaded S&P 500 10-K filings for fiscal years 2019-2025. Parsed HTML to extract MD&A sections using BeautifulSoup. Handled filing variants (10-K, 10-K/A, 10-K405) and missing MD&A cases.

Sentence Segmentation: Split MD&A text into ~1.96 million sentences using spaCy. Applied preprocessing rules to handle tables, exhibits, and boilerplate text.

AI Keyword Pre-filtering: Implemented regex-based filter to identify AI-relevant sentences. Reduced candidate pool from ~1.96M to ~3,580 sentences (0.18% hit rate). Keywords included: artificial intelligence, machine learning, deep learning, neural network, generative AI, large language model, automation, algorithm.

LLM Classification: Two concurrent pipeline variants:
- Kimi CLI subprocess variant: Spawns kimi CLI processes via subprocess. ThreadPoolExecutor with MAX_WORKERS=4. Each worker processes one sentence. Output parsed from stdout with Unicode bullet prefix detection.
- Generic OpenAI-compatible variant: Standard HTTP client with ThreadPoolExecutor(max_workers=8). Checkpoint/resume via incremental CSV append.

Classification taxonomy (mutually exclusive):
- Generic AI Disclosure: Mentions AI without specific implementation details
- Substantive AI Implementation: Describes actual AI deployment (technology, use case, investment)
- Substantive AI Risk/Governance: Discusses AI-related risks, ethics, governance frameworks
- AI Candidate: Contains AI keywords but insufficient context for classification

Output: firm_year_ai_disclosure_summary.csv with 5 intensity measures:
- n_ai_candidate_sentences
- n_generic_ai_disclosure_sentences
- n_substantive_ai_implementation_sentences
- n_substantive_ai_risk_governance_sentences
- n_substantive_ai_disclosure_sentences (aggregate)

Plus per-10,000-word normalized versions and sentence intensity ratios.

Slide 3: Critical Data Cleaning Challenges & Fixes

Challenge 1: Ticker Mismatch Between NLP and Compustat
- NLP uses hyphens: BRK-B, BF-B
- Compustat uses dots: BRK.B, BF.B
- Fix: Pre-merge ticker normalization map
  {'BRK-B': 'BRK.B', 'BF-B': 'BF.B', 'GOOG': 'GOOGL', 'NWS': 'NWSA', 'FOX': 'FOXA'}

Challenge 2: Fiscal Year Misalignment for Non-December FYE Companies
- Problem: NLP fiscal_year (filing year) != Compustat fyear for Jan-May FYE companies
- Example: BF-B FY2019 (May 2018-Apr 2019) has datadate=2019-04-30 but Compustat fyear=2018
- Naive merge on fiscal_year == fyear matches to wrong year's data (2-year lookahead bias)
- Affected: 42 tickers (~8% of S&P 500), 269 rows
- Fix: For Jan-May FYE companies, adjust merge key: controls_fyear = NLP_fiscal_year - 1
  Implementation: Parse datadate month, apply fyear_adj = fiscal_year - 1 when fye_month in [1,2,3,4,5]

Challenge 3: Dual-Class Share Duplicates
- Problem: Same 10-K filed under two tickers (GOOG/GOOGL, FOX/FOXA, NWS/NWSA)
- Creates duplicate accession_numbers, double-counting in regressions
- Detection: groupby(accession_number).size() > 1
- Fix: Keep primary class (has gvkey/cik in Compustat), drop secondary
  GOOG -> keep GOOGL, FOX -> keep FOXA, NWS -> keep NWSA
  21 duplicate accession numbers resolved, 21 rows dropped

Challenge 4: gvkey Zero-Padding in WRDS PostgreSQL
- Problem: WRDS stores gvkey as VARCHAR(6) with leading zeros ("001690")
- Python pandas reads as integer (1690) or float (1690.0)
- Passing "1690" to psycopg2 IN clause matches nothing for 5-digit gvkeys
- Fix: Always apply str(int(float(x))).zfill(6) before WRDS queries
- Impact: Without fix, supplemental variable coverage dropped from 98% to ~50%

Challenge 5: CRSP-IBES Linkage for Dual-Class Shares
- Direct ticker merge fails for dual-class shares (different ticker conventions)
- Correct path: gvkey -> permno (CCM) -> IBES ticker (ibcrsphist)
- Implementation: 
  Step 1: crsp.ccmxpf_lnkhist for gvkey-permno mapping
  Step 2: wrdsapps_link_crsp_ibes.ibcrsphist for permno-IBES_ticker mapping
  Step 3: ibes.statsum_epsus for consensus forecasts

Challenge 6: Missing gvkey/cik for Hyphenated Tickers
- BF-B and BRK-B had all gvkey/cik = NaN after initial merge
- Root cause: Controls CSV uses BF.B and BRK.B, NLP uses BF-B and BRK-B
- Fix: After ticker mapping, fill gvkey/cik from controls lookup
  BF-B: gvkey=2435, cik=14693
  BRK-B: gvkey=2176, cik=1067983
